# AIsa 文档系统 · 工作规划

**写给谁看：** 招聘方（Developer Advocate / 文档负责人评委），以及接手这套系统继续推进的我自己。
**读完能做什么决定：** 判断这套作品是不是「一个能自我维护的文档系统」，而非一篇一次性的样稿；并看清它下一步往哪走。
**一句话主线：** 已经把「检测漂移」做成自动机制，接下来要把「修复漂移」也交给 Agent，最后让整套系统对外可见、可复制。

---

## 一、主线：三步走

文档系统的演进只有一条主线，三个阶段，逐级把人从重复劳动里替换出来：

| 阶段 | 系统能力 | 人的角色 | 状态 |
| --- | --- | --- | --- |
| 第一阶 · 看见 | 实时抓取线上契约，自动判定漂移 | 看报告、定优先级 | ✅ 已交付 |
| 第二阶 · 修复 | 检测到缺口后自动生成译文 / 修复 PR | 审稿、合并 | 🔜 进行中 |
| 第三阶 · 放大 | 对外内容 + 专项技能，让系统可见、可复制 | 定方向 | 📋 已排期 |

判断标准始终是同一个：**这件事做完后，人还需不需要手动盯着做一遍？** 答案为「不需要」才算闭环。

---

## 二、归因拆解：四条工作线

把主线拆成实际要推进的四条线。每条都标注它解决主线的哪一段、当前到哪、下一步动作。

### 线 1 · 检测（已交付，持续运转）

- **已有：** `aisa_doc_auditor` 审计 Agent + `doc-reality-check.yml` CI 工作流 + `tools/docs_mirror.py` 全量镜像。
- **能力：** 每个 docs PR 拦截漂移并评论；每 6 小时定时扫描，把无 PR 的上游漂移转成 `doc-drift` Issue;镜像工具可随时把全部文档拉到本地，支持增量与 diff。
- **下一步：** 保持线上跟随。审计对象全为公开资源，结论随每次运行自动刷新，无需维护快照。

### 线 2 · 自动翻译闭环（进行中，最高价值）

- **要解决：** 中文覆盖率是当前最严重的阻断项（具体数字以审计实时结论为准），而中国是这个网关主打的供给市场。靠人逐页翻译追不上文档增长。
- **目标形态：** 把审计 Agent 升级成 **「检测 → 翻译 → 提 PR」全自动闭环**——审计发现某页缺中文版，即调用 AIsa 自己的模型（吃自家狗粮）生成译文，按本地化目录规范落盘，自动开一个翻译 PR 交人审稿。
- **关键设计点：**
  - 译文带 `source_of_truth` 与 `last_synced` 头，本身即可被审计 Agent 回查，避免译文反过来腐烂。
  - 用 AIsa 自家模型翻译，既是真实接入示范，也是岗位「吃狗粮、以外部开发者视角发现坑」的直接体现。
  - 先覆盖最高价值页（快速开始、核心 API、国产模型目录），不追求一次铺满。
- **依赖：** 需要一个 API Key 跑通线上翻译调用（评审复现时可用占位说明）。

### 线 3 · 对外内容（已排期）

- **要解决：** Developer Advocate 的产出不止代码，还要有面向社区、能带来开发者的对外内容。
- **两件交付物：**
  1. **命令行演示**：录一段 asciinema / 视频，展示「跑审计 → 看见漂移 → 自动提修复 PR」的完整链路，直观证明系统在运转。
  2. **技术文章**：一篇可发在掘金 / 知乎的长文，讲「如何用 AI Agent 让开发者文档不再腐烂」，以本作品为案例，定位为对外内容样例。
- **下一步：** 第二阶闭环跑通后即可录制，演示素材直接取自真实运行。

### 线 4 · 专项技能（已排期，待配置）

后续套用专属技能推进两件事，先记入规划，技能配置就绪后启动：

- **质量检查：** 对文档做系统性质检——死链、代码示例可运行性、术语一致性、结构完整性，补齐审计 Agent 之外的内容层质量门禁。
- **API 文档内容加强：** 针对 API 参考补全示例、参数说明、错误码、边界条件，把「能跑通」抬升到「好用、好懂」。

## 三、单点下钻：为什么第二阶是整盘棋的支点

四条线里，**自动翻译闭环是把作品从「检测器」升级为「自维护系统」的那一步**，值得单独放大看。

第一阶只解决了「看见」：它能告诉你中文覆盖率有多低，却仍要人去把每一页翻出来。只要修复还压在人身上，文档增长一快，缺口就会重新拉大——审计越勤奋，人越疲于奔命。这是「检测」类工具的天花板。

第二阶要打穿的正是这层天花板：让系统在发现缺口的同一时刻，就把修复动作生成出来。一旦「检测」和「修复」接上，这套东西才真正符合岗位要求的那句话——**让 AI Agent 融入文档开发业务系统、自动运转**，而不是又一个需要人喂的仪表盘。

它也是对岗位最有力的回答。用 AIsa 自家模型翻译自家文档，同时完成三件事：补上离收入最近的中文缺口、示范真实接入、以外部开发者视角验证 DX。这一个动作，就把「我能写文档」彻底讲成了「我能把文档变成系统」。

因此排序很清楚：**先打通第二阶闭环，它跑通了，第三阶的演示和文章有了真实素材，第四阶的专项技能也有了可挂载的底座。** 这是全盘的支点，不是四件并列任务里的一件。

---

## 四、里程碑顺序

1. **M1 · 检测线持续运转** — 已交付，维持线上跟随。
2. **M2 · 翻译闭环跑通** — 审计触发 → 调 AIsa 模型译最高价值页 → 自动提 PR。（支点）
3. **M3 · 对外内容** — 基于 M2 真实链路录命令行演示 + 写技术文章。
4. **M4 · 专项技能** — 质量检查、API 文档内容加强，待技能配置就绪后启动。

# 报告契约 · 最终交付物的形态标准（权威）

> 本文件规定 AIsa 文档质量工程**对外交付的两份报告**长什么样：必备骨架、证据字段、可追溯规则、验收口径。
> 它是「形态」的权威源，与「文字品味」的权威源 `../WRITING-STANDARD.zh-CN.md` 并列——后者管怎么写好一句话，本文件管一份报告该有哪些块、每块必须带什么证据。
> 任何被派去整理最终报告的 agent，先读本文件，再读 `WRITING-STANDARD.zh-CN.md`，二者冲突时本文件管结构、写作规范管文风。

---

## 一、为什么要有契约：先想清楚受众和他的那一个问题

报告的读者是 **产品负责人**。他时间稀缺，要的不是「我们干了多少活」，而是「文档现在到底什么状况、根因是什么、我该让谁去改什么」。

我们调研过世界级咨询公司（麦肯锡、BCG、Bain）怎么向高管交研究结论，提炼出五条硬规则，本契约据此设计：

1. **结论先行（金字塔原理）**：先给答案，再给支撑论点，最后才是数据。高管可能只看前两屏[[The Pyramid Principle: How Consultants Structure Arguments That Win]](https://deckary.com/blog/pyramid-principle-consulting)。
2. **SCQA 开场**：情境（现状，对方已认同）→ 冲突（什么变了/坏了）→ 问题（自然引出的那一问）→ 答案（你的结论），用它搭执行摘要的第一段[[SCQA Logic]](https://thinkinsights.net/strategy/scqa-logic)。
3. **结论式标题（action title）**：每个章节标题是一句结论，不是一个话题。写「文档把 645 路径的 spec 描述成 121 schema」，不写「schema 数核对」——让人只读标题就能串起全局[[How to Write Slide Action Titles Like McKinsey]](https://slideworks.io/resources/how-to-write-action-titles-like-mckinsey)。
4. **发现 ≠ 洞察，必须答「so what」**：「churn 涨了 5%」是发现，「因竞品调价导致、故应调整忠诚度计划」才是洞察。每条问题都要落到「会坑到谁、该怎么根治」[[The McKinsey Way: How to Present Research Findings]](https://tosea.ai/blog/mckinsey-way-present-research-findings)。
5. **靠透明建立可信**：主动交代口径与边界（撤回了什么、哪些待人工确认、为什么不报），不藏短板。这反而增强权威，并堵住对方挑刺的入口[[The McKinsey Way: How to Present Research Findings]](https://tosea.ai/blog/mckinsey-way-present-research-findings)。

本项目自己最看重的一条，与第 4 条同源：**发现一个问题不重要，找到它的根本原因、能从源头杜绝它才重要。** 两份报告的重心都压在根因与建议上，不停留在「列了多少处错」。

## 二、两份报告，两条叙事线

阶段一（硬伤）与阶段二（散文 AI 友好度）本质是同一件事——**对存量文档的一次内容审计**，合成一份《报告甲》。阶段三是另一件事——**在已查清的 API 生产管线上提供加强能力**，单出《报告乙》。

| | 报告甲：存量文档内容审计 | 报告乙：API 内容加强 |
| --- | --- | --- |
| 覆盖 | 阶段一 + 阶段二 | 阶段三 |
| 一句话主张 | 文档在「发现入口」与「跨页事实」两处系统性失真，根因是缺单一权威事实源 | 线上 API 参考正文由 spec 内联渲染，我们在这条管线上提供可对账的双字段加强能力 |
| 叙事线 | 审计对象 → 关键发现（生产模式 / 统计 / 问题模式与根因）→ 怎么做的 → 具体审计报告 → 建议 | 发现的 API 生产管线 → 管线上能提供的加强能力 → 能力的范畴与价值 → 典型实例效果 → 完整产物 |
| 何时可写 | **现在即可**（阶段一、二产物已齐） | **须等阶段三 v2 执行产出后**（见第六节依赖） |

## 三、《报告甲》必备骨架

按金字塔原理自上而下，章节标题一律写成结论句。

**0. 标题与元信息**：审计对象、范围（页数构成）、日期、源仓库与 pinned commit、一句话方法、判定红线（零误报）。

**1. 执行摘要（结论先行，独立成章可单读）**
- 用 SCQA 开场：现状（文档同时面向人和 Agent）→ 冲突（发现入口与跨页事实失真）→ 问题（Agent 会不会被带偏、根因在哪）→ 答案（一句话主结论）。
- 给 3–5 条 MECE 支撑论点，每条一句话带最关键的一个数。
- 点名**单一根因**与**最高优先级建议**。
- 全章读完不超过一屏。

**2. 关键发现**（本报告的「维度 1 + 维度 2」）
- 2.1 **文档生产模式**：把已查清的链路（源仓库 `.mdx` + `openapi/*.json` → 部署层 → `aisa.one`；664 桩页正文由 spec 内联渲染、非手写）讲清，作为后续发现的背景。
- 2.2 **AI-Friendly 专项审计统计结果**：规模盘（743 页 / 664 桩页 / 73 散文页 / 31 spec / 11 中文页）+ 阶段一 3 项 BLOCKER + 阶段二 20 条缺口（7 高 / 8 中高 / 5 中）+ 去向台账（47 原始 = 20 保留 + 21 并入 + 6 撤回）。
- 2.3 **问题模式与根因**：把全部缺口归到一层病灶（**只拆一层**），点明同一根因：agent-skills 各页手抄事实、不引用权威目录；agent-discovery 是固化的手写快照、未随 spec 演进。

**3. 我们如何做的**（方法 + 可信度）
- 取数：源仓库、pinned commit、镜像，强调「数字会变、复现命令是不变真相」。
- 工具：阶段一确定性静态检查器 `checks/src_consistency.py`（零依赖、可 `--ci`）；阶段二 Map-Reduce 多 agent（三路探查 + 单汇总、三层隔离防确认偏误）。
- **可信度口径（透明即权威）**：两条硬线撤回律、逐字复核、去向台账可对账、权威源单一化；如实交代撤回的 6 条与待人工确认的 5 处。

**4. 具体审计报告**（详表，本报告的可消费核心，见第五节字段规范）
- 一张缺口登记表，每条带齐第五节全部证据字段；BLOCKER 与散文缺口可分两段，但字段口径统一。
- 不得简陋：不能只写「某页数字不对」，必须让读者点开 URL、跳到 git 对应行、Ctrl+F 命中原文、跑一条命令拿到真值。

**5. 我们的建议**（根治，按优先级，本报告的「维度 2.5」落点）
- 最高优先级：为 agent-skills 建立对 `guides/models.mdx` + 定价页的**单一引用**，删除离群价格/计数/端点/旧型号——14 条 6.1 随之消解。
- 次优先级：让 `agent-discovery.mdx` 的数字与分类表由 `openapi.yaml` **自动派生**，杜绝再漂移。
- 收尾：两处单篇算不平的具体改法。
- 每条建议写明「改完之后哪些缺口一次性消失」，即根治效果。

**6. 附录**：全部复现命令清单、3 项 BLOCKER + 20 条缺口完整明细、撤回/待确认台账。

## 四、《报告乙》必备骨架

**0. 标题与元信息**：覆盖的 spec 数 / operation 数 / 字段数、日期、commit、两道闸门口径。

**1. 执行摘要（结论先行）**：一句话——线上 API 参考正文是 spec 内联渲染产物，我们在这条管线上以 `x-doc` 命名空间只增不改地做**双字段加强**（加强英文 `desc_en` + 中文本地化 `title_zh`），写进增强 spec `enhanced.json`，并用两道确定性闸门（原生保全 exit 0 + 三栏对照表完整性 exit 0）证明「只增不改、每个对外字段都改到位」。

**2. 发现的 API 生产管线**：spec → `scripts/consolidate_openapi.py` → 线上内联渲染；点明 v1 的教训（火山 `direct` 渲染器按 `Action`/`Version` 信封模板生成、不渲染真实 REST 参数，故**不以渲染 md 为交付**——权威呈现改用可抽取的三栏对照表，见 D-015）。

**3. 管线上提供的加强能力**：`x-doc` 命名空间加性注入（只增不改、不 hoist、不碰原生结构）→ 双字段增强内容 `content.json` → 增强 spec `enhanced.json`（原生零改动）→ 三栏对照表 `review_table.md`（源英文 → 加强英文 → 中文，覆盖接口描述 + 参数 + 请求/响应字段）；两道确定性闸门把关——`check_native_preserved.py` 原生保全（exit 0）+ `make_review.py` 对外字段完整性（每字段须齐备 `desc_en`+`title_zh`，缺漏即非零）。

**4. 能力的范畴与价值**：覆盖了哪些 spec / 哪些字段类型；加强了哪几类内容（用途、取值业务含义、依赖、坑、错误码）；价值口径——双字段（英文加强 + 中文本地化）语义一致、只增不改、`source=human` 永不被 AI 静默覆盖、可逐字段对账、不注水、不复制原生元数据。

**5. 典型实例效果展示**：从三栏对照表挑代表性字段，展示「源英文 → 加强英文 → 中文本地化」三态对比（含一处深层内联 / `oneOf` 字段，证明加强触达嵌套结构）；至少含一处「原生保全 diff 为零」的证明、一处「AI 不静默覆盖人工(分流进 review)」的演示。

**6. 完整产物附录**：每个 spec 的 `content.json` / `enhanced.json` / `review_table.md` + `.json` / `inject_review.json` 清单（**无 md 渲染页**）+ 两道闸门状态总表 + 与 v1/D-013 试点的差异与改进。

## 五、证据字段规范（《报告甲》详表每一条都必须带齐）

这是「站在使用者视角、好查证好理解」的硬性落地。缺任一字段即不合格。

| 字段 | 内容 | 取值规范 |
| --- | --- | --- |
| 编号 | B1/B2… 或 P1/P2… | 沿用现有报告编号，全篇唯一 |
| 严重度 | BLOCKER / 高 / 中高 / 中 | 与现有报告一致 |
| 文档位置 | 文件相对路径 + 行号 | 如 `guides/agent-discovery.mdx:16` |
| **线上文档 URL** | 该页线上地址 | `https://aisa.one/<mdx 相对路径去掉 .mdx>`，最终以 `docs.json` 的 navigation 为准；派遣 agent 抽样打开 1–2 个自验，不可凭空构造 |
| **git 源地址** | 指向具体行的 permalink | `https://github.com/AIsa-team/docs/blob/16863d3/<路径>#L<行号>`，**用 pinned commit `16863d3`**（审计时 HEAD），保证行号永久稳定，不用 `main`（会漂移） |
| 问题描述 | 一句话说清「错在哪、会坑到谁」 | 说人话，落到「Agent 会因此做错什么」 |
| 证据引用 | 文档原文摘录 | 可被 Ctrl+F 在源文件命中的原文片段 |
| 真值 / 复现 | 正确值 + 一条可独立跑的命令 | 如 `python3 -c "...openapi.yaml...schemas..."` → 321 |
| 根因归类 | 病灶类型 + 规范点号 | 如「同模型几套价 · 6.1」 |
| 建议动作 | 这条该怎么根治 | 指向第五节建议里的对应项 |

**可消费性硬规则**：
- 每条都要能「点 URL 看线上 → 点 git 跳到行 → Ctrl+F 命中原文 → 跑命令验真值」四步闭环，任一步断链即不合格。
- 复现命令必须在源仓库根目录可独立运行、确定性输出，不依赖任何 live 请求（live 仅作旁证，沿用 D-005）。
- git permalink 一律钉在 `16863d3`；如重跑时 HEAD 已变，须在元信息里同时给出新 commit 并重算行号。

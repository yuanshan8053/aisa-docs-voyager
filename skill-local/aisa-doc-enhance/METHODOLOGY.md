# 写作方法论 · AIsa OpenAPI 双字段加强（唯一权威）

> 本文件是「阶段三·LLM 推理加强」唯一的写作内核，从 `api-doc-agent` 的 `doc-engine/core/writing_methodology.md` + `style_rules.md` + `x-doc.spec.md` 蒸馏而来，已彻底去平台化、去引擎化。**派遣的 agent 只读本文件 + 本目录 `SKILL.md`，不依赖任何包外引擎。**
>
> 写作目标：站在 API **使用者**视角，把 spec 里只有机器结构的字段，补成「这个字段干什么、怎么用、有什么坑」的人话——分两层产出：**加强英文 `desc_en`** 与**中文本地化 `title_zh`**。

---

## 0. 双字段模型（本方法论的核心，逐字段都要产）

每个对外字段/参数/接口，产两层文档内容，挂进 `x-doc`：

| 层 | 字段 | 来源 | 要求 |
|---|---|---|---|
| 源英文 | 原生 `description`（**只读，绝不动**） | spec 自带 | 作为对照基准保留 |
| **加强英文** | `x-doc.desc_en` | 在源英文基础上**加强** | 补全用途/取值含义/依赖/坑；源英文若已够好则精炼保留其意 |
| **中文本地化** | `x-doc.title_zh` | 把**加强后的英文**本地化 | 不是逐字翻译，是地道中文表达；与 `desc_en` 语义一致 |

**顺序铁律**：先想清楚「这个字段到底干什么」→ 写出加强英文 `desc_en` → 再据 `desc_en` 做中文本地化 `title_zh`。**两者必须语义一致**，不允许英文说一套中文说另一套。

---

## 1. 最高原则：描述深度【自适应字段复杂度】

凌驾于一切清单与模板之上：

- **简单字段一句话。** 纯 ID、布尔开关、自解释的名称/标题、时间戳——「是什么 + 一句关键提示」即可。对标 Stripe（客户 ID 只写「已存在客户的 ID」）。
- **复杂字段才展开。** 仅当字段确有单位/取值范围/多枚举语义/跨字段依赖/默认行为时，才逐项展开。
- **六要素是「想全」的启发清单，不是填空题。** 见第 3 节，能答几项答几项，无关的略过。
- **红线·禁止注水。** 给一个纯 ID 硬凑「从哪获取/何时生效/如何组合/不传会怎样」四段空话，是**降低**质量。验收专项抽查注水。

> 五条质量锚（自检）：① 对开发者有帮助；② 有全局线索（主动关联依赖/互斥/获取接口）；③ 一看就懂、一用不出错；④ 语言自然流畅；⑤ 专业可信。

---

## 2. 先理解 API，再动笔

通读 spec：核心资源是什么、操作类型（创建/查询/列表/删除/搜索…）、典型调用链、关键依赖字段（资源 ID、游标、token、回调 URL）、请求字段与响应字段的**读写方向**（别把响应含义写成请求说明）。

- **接口级 `desc_en` / `description_zh`**：1–2 句，说清**接口能力 + 关键配置 + 典型场景**。不要把逐字段解释塞进接口描述。
- **接口级 `heading_zh`**：文档 H1，形如「中文动作 + Action 名」（如 `创建对话补全 createChatCompletion`）。动作用「动词 + 资源」概括，Action 名照搬 `operationId`，中间一个半角空格。

---

## 3. 六要素启发清单（按需回答，简单字段往往只需第 1 项）

1. **传什么**：字段值的业务含义，不只翻译字段名。
2. **为什么传**：影响什么能力/范围/路由/筛选/排序/返回。
3. **从哪获取**：ID/token/游标/地址等说明**来源接口**（优先引用相关 `operationId`）。
4. **何时生效**：依赖的开关/策略/资源状态/地域/版本。
5. **如何组合**：必须同时设 / 二选一 / 互斥 / 成组 / 数组限制 / 字段联动。
6. **不传会怎样**：默认行为；**未知时转批注**，不编造默认值。

---

## 4. 两条边界（极易踩坑，必须遵守）

**边界一·不复制原生元数据。** 字段名 / 是否必选 / 类型 / 枚举字面量 / 取值范围 / 默认值，都是 OpenAPI 原生结构，渲染器从原生读取，**严禁复制进 `desc_en` / `title_zh`**（制造双份真相）。枚举的**业务含义**可写（`0` 表示什么场景），但不要写「该字段类型是 String、是否必选为是」。

**边界二·不动原生 `description`。** 源英文是对照基准，永远保留在原生 `description`，加强后的英文写进 `x-doc.desc_en`，二者并存可对账。绝不把加强内容覆盖回原生 `description`。

---

## 5. 按字段类型选骨架（起点，仍受第 1 节自适应深度约束）

| 类型 | 英文骨架 → 中文骨架 |
|---|---|
| **ID 类** | `Unique identifier of <resource>. Obtainable via <operationId>, used to <action>.` → `<资源>的唯一标识。可通过 <operationId> 获取，用于 <动作>。` |
| **名称类** | `Name of <resource>, used to identify it within <scope>.` → `<资源>名称，用于在 <范围> 内识别。` |
| **描述类** | `Free-text description of <resource>.`（一句话） → `<资源>的描述。` |
| **开关类** | 解释 `true`/`false` 各自行为差异，不复述原生 default |
| **枚举类** | 逐枚举值解释**业务含义**（字面量走原生 enum） |
| **时间类** | 点明格式（Unix 秒 / ISO 8601）、时区、窗口 |
| **数值类** | 单位 / 范围（可溯源才写）/ 超限行为 |
| **凭证类** | 用途 + 脱敏存储、避免明文日志、轮换建议 |
| **分页类** | 页码/页大小/游标来源/最大值 |

类型识别：名含 `ID`/`Id` 或 `$ref` 指资源→ID 类；`Name`→名称类；`type:boolean` 或名含 `Enable`/`Switch`→开关类；有 `enum`→枚举类；名含 `Time`/`At`/`Date`→时间类；`Key`/`Secret`/`Token`→凭证类；`Page`/`Cursor`→分页类。

---

## 6. 风格（中文本地化 title_zh）

- 字段名 / 枚举字面量 / `operationId` / 路径 **保留英文原文**；代码字面量用反引号 `` ` ``。
- 中文用全角标点；CJK 与英文/数字之间留 1 个半角空格。
- 避免空话（「请求参数」「响应字段」这种无信息量短语）。
- 剔除源码痕迹（`Args:` / `:param` / `TODO`）。

---

## 7. 不捏造 → 转批注

拿不准的硬事实（范围/单位/枚举含义/默认值/依赖接口名）一律转批注，**绝不编造**。在 content.json 该字段写 `"annotation": "源码未声明取值范围，待研发确认。"`，注入器会自动以固定格式 `\n[⚠️批注:...]`（中文）/ `\n[⚠️Note:...]`（英文）内联进 `title_zh` / `desc_en`。

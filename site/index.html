项目历程：先看清，再分流，各个击破

起点是对文档一无所知。整体节奏是「看清文档怎么生产 → 按现状把对象分流 → 两条线分别治理 → 最后呈现与交付」。

1

摸清文档生产模式

源仓库 AIsa-team/docs（.mdx 散文 + openapi/*.json）经部署层渲染成线上 aisa.one——给人看 HTML，给 Agent 看 .md 孪生页与发现契约。把 GitHub 源与线上两份都抓到本地比对，推断出完整生产链路。

2

现状统计，据此分流

743 页 .mdx 中，664 页是 API 参考桩页（正文由 spec 内联渲染，非手写），真正可做散文质检的只有 73 页人写散文。两类状态与风格迥异，于是分两条线治理。

3

手写散文 → 质检

阶段一静态硬伤检查（零依赖、零误报），阶段二对 73 页做 AI 友好度系统质检。20 条缺口收口到一个根因：agent-skills 缺一个被全站引用的权威事实源。

4

API 文档 → 加强 · 翻译 · 上线

源仓库全部 31 份 spec 逐字段加强英文描述 + 中文本地化，加性注入 x-doc（原生零改动），投影成干净标准 OpenAPI；其中 30 份 AIsa 真实 spec（剔除 1 份玩具示例）由 Scalar 渲染成双语站点上线 GitHub Pages。

这套工程怎么守住质量

文档负责人的标准，落到每一条可机检的纪律上。

零误报是红线

凡列为阻断硬伤的，必须仅凭源仓库文件即可 Ctrl+F 证伪；live 探测只作旁证，绝不单独定罪。曾据此主动撤回一条无法证伪的误报。

绝不杜撰，主动留痕

源 spec 没声明清楚的硬事实一律不编，转成显式批注 [⚠️待研发确认]。109 条批注就是这条红线的落地——诚实留白比自信错误更有价值。

原生零改动，只增不改

加强成果加性挂在 x-doc，原生 description/summary 逐字节不动。check_native_preserved.py 对 31 份 spec exit 0 机器坐实。

一切可复现

每个数字都附可独立跑的命令，基线钉死 pinned commit 16863d3。python3 ws-v2/central_accept.py 一键复跑全量验收。
